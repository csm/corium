# corium-wasm

An **ephemeral, in-memory corium database compiled to WebAssembly**, with a
curated MusicBrainz dataset of 1997 releases embedded. A fully queryable — and
transactable — subset of corium that runs entirely in a browser tab, with
nothing but the pure engine crates underneath. Reload the page and it resets.

```js
import init, { Mbrainz } from "./corium_wasm.js";
await init();
const mb = new Mbrainz();

// EDN Datalog, executed peer-locally against the current Db value:
mb.q('[:find ?name ?year :where [?r :release/year ?year] [?r :release/name ?name]]');

// Transact into the in-memory value; visible to the next query, gone on reload:
mb.transact('[{:db/id "x" :artist/gid #uuid "…" :artist/name "New Artist"}]');

mb.stats();   // {:basis-t … :transactions … :datoms …}
```

Results and reports are returned as **EDN strings**.

## What's inside

No transactor, peer, gRPC, store, or log — none of the async/networked stack.
Just the pure engine subset, which has no `tokio`, filesystem, socket, or clock
dependencies and compiles to `wasm32-unknown-unknown` as-is:

| Crate | Role |
|---|---|
| `corium-core` | values, datoms, ids, schema model |
| `corium-db` | the immutable `Db` value + covering indexes |
| `corium-tx` | transaction expansion / tempid + lookup resolution |
| `corium-query` | EDN Datalog compiler, planner, executor (+ its own EDN reader) |
| `corium-forms` | EDN ⇄ schema/transaction-form conversion (shared with `corium-protocol`) |

The [`Engine`](src/engine.rs) type is pure Rust and is unit-tested on the host
toolchain (`cargo test -p corium-wasm`); [`lib.rs`](src/lib.rs) is the thin
`wasm-bindgen` wrapper. `Engine::apply` mirrors
`corium_transactor::Transactor::transact` minus the durable log and wall-clock
instant, which are meaningless for a memory-only database.

## The dataset

`assets/releases-1997.edn` — **20 releases, 20 artists, 19 labels, 266 tracks**
— is generated by [`assets/gen_releases_1997.py`](assets/gen_releases_1997.py)
and embedded via `include_str!`, so the artifact is one self-contained `.wasm`
with no network fetch. `assets/schema.edn` mirrors
`examples/musicbrainz/schema.edn`.

**Data provenance:** artist, album, label, and track *names* are real; track
*durations* are approximate and the UUID `gid`s are synthetic (namespaced by
type, not the canonical MusicBrainz identifiers). The network policy in this
environment blocks `musicbrainz.org`, so a live extraction wasn't possible —
but the EDN shape matches the real loader output, so swapping in a real
extraction later (releases from any year, from the MB web service or data
dumps) is a drop-in file replacement. Re-generate with:

```sh
python3 assets/gen_releases_1997.py > assets/releases-1997.edn
```

## Building the web demo

The site lives in the repo-root [`web/`](../../web/) directory
(`index.html` landing page, `mbrainz.html` demo). The demo page imports the
wasm bindings from the same directory, so build them there. Needs the
`wasm32-unknown-unknown` target and
[`wasm-bindgen-cli`](https://crates.io/crates/wasm-bindgen-cli).

```sh
rustup target add wasm32-unknown-unknown
cargo install wasm-bindgen-cli          # matching the wasm-bindgen dep version

# from the repo root:
cargo build -p corium-wasm --target wasm32-unknown-unknown --release
wasm-bindgen target/wasm32-unknown-unknown/release/corium_wasm.wasm \
  --out-dir web --target web --no-typescript

# serve the site (module scripts need http, not file://):
python3 -m http.server -d web 8080
# open http://localhost:8080  (demo at /mbrainz.html)
```

Optionally shrink the artifact with `wasm-opt -Oz --enable-reference-types web/corium_wasm_bg.wasm -o web/corium_wasm_bg.wasm`.
Use a current [binaryen](https://github.com/WebAssembly/binaryen) (≥ v110 — e.g.
`npm install binaryen`, **not** Ubuntu's apt v108). Older binaryen has
[binaryen#4711](https://github.com/WebAssembly/binaryen/issues/4711): it
mis-points wasm-bindgen's `__wbindgen_externrefs` table export at the funcref
table, so the demo fails to init in every browser with `WebAssembly.Table.grow …
could not grow the table`.

The generated `corium_wasm.js` / `corium_wasm_bg.wasm` are gitignored;
`.github/workflows/publish-site.yml` rebuilds them and uploads `web/` over
SFTP on pushes to `main` (or via manual dispatch).

## Query examples

```clojure
;; every 1997 release
[:find ?name ?year :where [?r :release/year ?year] [?r :release/name ?name]]

;; OK Computer's tracklist, in order
[:find ?pos ?name
 :where [?r :release/name "OK Computer"] [?r :release/media ?m]
        [?m :medium/tracks ?t] [?t :track/position ?pos] [?t :track/name ?name]]

;; artists from Great Britain
[:find ?name :where [?a :artist/country :country/GB] [?a :artist/name ?name]]
```
